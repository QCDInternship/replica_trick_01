# Phase 6 Bottleneck Decision

## Goal

Phase 6 adds an integrated profiling suite after Phase 5 to measure where time
is now spent in a tiny end-to-end workflow. The decision question is whether
configuration generation is still the dominant bottleneck, or whether
measurement/action analysis has become important enough that the Phase 5
backend is already addressing a meaningful cost.

## Non-goals

- No changes to configuration generation.
- No changes to `markov_chain_sweep_replica`.
- No changes to proposal generation, RNG behavior, Metropolis acceptance,
  update ordering, replica boundary conditions, staple definitions, action
  definitions, tadpole-improvement logic, or save frequency.
- No partition-boundary caching.
- No CUDA generation.
- No parallel link updates.
- No GPU defaults.
- No removal of CPU reference paths.
- No physics-output changes.

## Profiling Commands

Run the default tiny end-to-end profile:

```bash
python scripts/profile_end_to_end_pipeline.py
```

Profile measurement/action paths only on an existing saved configuration:

```bash
python scripts/profile_end_to_end_pipeline.py --skip-generation --config path/to/config.npy
```

Request GPU measurement timing when CuPy/CUDA is available:

```bash
python scripts/profile_end_to_end_pipeline.py --use-gpu
```

Require GPU execution and fail if CuPy/CUDA is unavailable:

```bash
python scripts/profile_end_to_end_pipeline.py --use-gpu --strict-gpu
```

Use a custom output directory:

```bash
python scripts/profile_end_to_end_pipeline.py --output-dir profiles/phase6_end_to_end_custom
```

Generate the Phase 6 decision report from the default profile outputs:

```bash
python scripts/make_phase6_decision_report.py
```

Run synthetic microbenchmarks for candidate generation subkernels:

```bash
python scripts/microbenchmark_generation_subkernels.py
```

Run a short CPU-only smoke benchmark:

```bash
python scripts/microbenchmark_generation_subkernels.py --batch-size 100 --repeat 2
```

Request optional GPU microbenchmarks when CuPy/CUDA is available:

```bash
python scripts/microbenchmark_generation_subkernels.py --use-gpu
```

## Expected Outputs

The default command writes:

```text
profiles/phase6_end_to_end/profile.prof
profiles/phase6_end_to_end/profile_top.txt
profiles/phase6_end_to_end/timing_summary.json
docs/PHASE6_DECISION_REPORT.md
benchmarks/generation_subkernels.json
```

`profile.prof` is the raw `cProfile` data. `profile_top.txt` is a cumulative
time summary from `pstats`. `timing_summary.json` records environment metadata,
the workload, section timings, and timing fractions.
`docs/PHASE6_DECISION_REPORT.md` is generated by
`scripts/make_phase6_decision_report.py` and contains the go/no-go
recommendation for future CUDA configuration-generation investigation.
`benchmarks/generation_subkernels.json` is generated by
`scripts/microbenchmark_generation_subkernels.py` and records synthetic batched
math timings.

## Interpreting Timing Fractions

`fraction_time_generation` is:

```text
generation_time_seconds / total_end_to_end_time_seconds
```

`fraction_time_measurement_action` is:

```text
(cpu_reference_measurement_time_seconds
 + backend_cpu_measurement_time_seconds
 + backend_gpu_measurement_time_seconds if present)
/ total_end_to_end_time_seconds
```

If `fraction_time_generation` remains large, configuration generation is still
the primary bottleneck and Phase 5 measurement acceleration is not enough to
change the next optimization target. If `fraction_time_measurement_action`
becomes comparable to generation, then measurement/action analysis is now a
meaningful cost and Phase 5-style backend work has practical value.

The default workload is intentionally tiny. Use it to compare relative time
placement and profiler hot spots, not to claim production performance.

## Synthetic Subkernel Microbenchmarks

`scripts/microbenchmark_generation_subkernels.py` benchmarks isolated synthetic
operations only:

- batched complex `3x3` matrix multiplication
- dagger/conjugate-transpose
- real trace
- simple Metropolis scalar arithmetic
- a staple-like local product assembled from synthetic matrices

These microbenchmarks do not use production configurations, do not call
`markov_chain_sweep_replica`, and do not implement GPU Markov-chain generation.
CPU NumPy remains the reference path, and optional GPU timings use the Phase 2
`gpu_backend` only for synthetic arrays.

Microbenchmark results do not prove full generation speedup. GPU speedup on
batched math does not automatically imply safe GPU Markov-chain sweeps, because
the full generator is update-order sensitive and includes RNG sequencing,
Metropolis decisions, boundary conditions, and save behavior. Full generation
would still require a checkerboard/red-black design and physics validation
before any production change.
