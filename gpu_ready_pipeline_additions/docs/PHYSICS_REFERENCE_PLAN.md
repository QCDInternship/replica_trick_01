# Phase 0 Physics Reference Plan

Phase 0 is a documentation and inventory baseline only. It defines the current
physics workflow and the reference outputs that later CPU/GPU implementations
must reproduce. Phase 0 must not optimise, refactor, or alter the Markov-chain
algorithm.

## Active Generation Path

The root active replica generation path is:

1. `Modified_Replica_generate_multiprocessing` defines the production parameters:
   `action`, `Nt`, `Nx`, `Ny`, `Nz`, `beta`, `Nstart`, `Nend`, `Ncfg`, `Nhits`,
   `n`, `s`, `u0`, `alphas`, and `ensembles`.
2. `create_base()` builds the base output path:
   `RunReplica{run_number}/Replica_{action}_{Nt}x{Nx}x{Ny}x{Nz}_b{int(beta * 100)}`.
3. Each `(alpha, ensemble)` job calls `run_replica_simulation(...)`.
4. `run_replica_simulation(...)` loads `config_{Nstart}_U1.npy` from the job
   directory when present; otherwise it starts from identity links through
   `gauge_latticeqcd.ReplicaLattice`.
5. Proposal matrices are generated by `gauge_latticeqcd.create_su3_set()`,
   which builds SU(3) proposals from SU(2) matrices and includes each proposal
   followed by its Hermitian conjugate.
6. `ReplicaLattice.markov_chain_sweep_replica(...)` performs the replica Markov
   chain. For each generated configuration it loops over lattice sites,
   directions, and hits; computes staples for `xcutoff_1 = Nx - s` and
   `xcutoff_2 = Nx - (s + 1)`; applies the existing Metropolis acceptance rule;
   updates accepted links in place; and saves `config_{cfg}_U1.npy`.
7. Analysis functions load saved configurations, compute `S_L`, `S_{L+1}`,
   `S_int`, action differences, alpha averages, Simpson integration, Sommer
   scale, and the final `C(l)` observable.

The ordinary non-replica lattice generation path is in `gauge_latticeqcd.generate`
and `gauge_latticeqcd.lattice.markov_chain_sweep(...)`. It uses the same
proposal generator and records ordinary Wilson-action configurations as
`link_{name}_{cfg}` when a save name is provided.

## Tadpole-Improvement Path

The active tadpole-improvement path is under `replica_trick_01/tadpole_improvement/`.

1. `Modified_Replica_generate_multiprocessing.py` defines tadpole-enabled
   parameters, including `USE_TADPOLE = True` and a placeholder `u0`.
2. `calibrate_u0()` loads `u0_cache_{Nt}x{Nx}x{Ny}x{Nz}_b{int(beta*100)}.txt`
   when available.
3. If no cache is available, `tadpole.solve_u0_selfconsistent(...)` estimates
   `u0` from the bare average plaquette using the Lepage-Mackenzie prescription.
   The calibration uses the non-replica lattice class because `alpha=0` in the
   replica class still applies replica temporal boundary handling.
4. The calibrated or cached `u0` is passed into
   `gauge_latticeqcd.ReplicaLattice`.
5. The tadpole-improvement `gauge_latticeqcd.py` applies the same replica
   generation workflow, with `u0` factors included in staple/action formulas.
6. Tadpole analysis computes the same reference quantities as the root path and
   includes `u0` in plot annotations.

Future implementations must preserve cached-or-measured `u0` behaviour when
tadpole improvement is enabled.

## Reference Observables To Preserve

Future CPU/GPU implementations must preserve the following observables and
diagnostics where they are available in the current workflow:

- Average plaquette history from `plaq_v_cfg.py` or equivalent calls to
  `gauge_latticeqcd.fn_average_plaquette(...)`.
- Ordinary Wilson action history from `action_v_cfg.py` or
  `gauge_latticeqcd.fn_eval_point_S(...)`, for non-replica configurations.
- Replica action from `ReplicaLattice.calc_action_replica(...)`.
- `S_L`, computed with `xcutoff_L = Nx - s`.
- `S_{L+1}`, computed with `xcutoff_Lp1 = Nx - (s + 1)`.
- `S_int = (1 - alpha) * S_L + alpha * S_{L+1}`.
- Action difference `(S_{L+1} - S_L) / (Ny * Nz)` as computed by
  `compute_action_diff(...)`.
- Alpha-averaged action differences and their standard deviations from
  `analyse_action_differences(...)`.
- Integrated action difference from `integrate_action_diffs(...)`.
- Sommer scale and final entropy-related `C(l)` observable from `c_func(...)`.
- Cached or measured `u0` when tadpole improvement is enabled.

Acceptance ratio should be recorded only if it is already exposed by the
current generator output. Phase 0 must not modify the generator to expose,
return, save, or aggregate acceptance ratios.

## Validation Rules

- Exact equality is not required for independent Markov chains unless the same
  RNG path, proposal sequence, update ordering, and initial configuration are
  explicitly preserved.
- Numerical helper rewrites must satisfy stricter checks, such as
  `np.allclose(...)` or absolute differences below `1e-12`, against the current
  Python formulas on fixed inputs.
- Algorithmic or implementation changes must be judged by preserved observables
  and uncertainty/error bars, not by bitwise equality between statistically
  independent Markov chains.
- Any future validation report should record the run parameters, RNG handling,
  thermalization cut, configuration range, `u0` source, and observable
  uncertainty estimates.

## Tiny Reference Run

Use `scripts/make_physics_reference.py` to create a tiny CPU reference dataset
with the active implementation. The script saves generated configurations under
`reference_outputs/` and writes small JSON summaries that can be compared by
future implementations.

Example:

```sh
python scripts/make_physics_reference.py --output-dir reference_outputs/phase0_tiny --seed 12345 --n-cfg 5
```

The generated `.npy` configuration files under `reference_outputs/` are ignored
by Git. The JSON metadata and observables summaries are intentionally small and
may be committed when useful.

## Reference Validation

Use `scripts/validate_against_reference.py` to compare future observables JSON
files against the Phase 0 reference.

Example:

```sh
python scripts/validate_against_reference.py --reference reference_outputs/phase0_tiny/reference_observables.json --candidate reference_outputs/new_run/reference_observables.json --mode strict
```

Use `strict` mode for implementation-only rewrites where the same inputs,
RNG path, proposal order, update order, and saved configurations are expected
to produce numerically identical or nearly identical observable values. This is
appropriate for helper rewrites, serialization changes, or other changes that
should not alter the Markov-chain trajectory.

Use `statistical` mode for independent Markov chains or implementations where
the RNG trajectory is not expected to match exactly. This mode compares
histories through means and standard deviations with looser tolerances. It is
only a screening check and is not a replacement for full physics/statistics
validation.

Strict equality is not expected after independent Markov-chain changes because
small differences in RNG streams, proposal order, or accepted updates can lead
to different valid configurations. Even so, the physics observables and their
uncertainties must be checked before trusting CPU, MPI, Numba, or GPU-ready
code.
